LeetCode 146：LRU 缓存

设计一个缓存：最近使用过的数据优先保留，容量满时淘汰最久未使用的数据。关键是让 get 和 put 都达到平均 O(1)。

Java HashMap 双向链表 设计题
O(1) get / put 平均时间复杂度

题目概述

实现 LRUCache 类，支持两个操作：

  • get(key)：如果 key 存在，返回 value，并把它标记为最近使用；否则返回 -1。
  • put(key, value)：插入或更新 key。如果容量超限，淘汰最久未使用的 key。

LRU 是什么

Least Recently Used，最久未使用。每次访问或写入一个 key，这个 key 都变成“最近使用”。容量满时，删除“最久没被碰过”的 key。

操作 返回 缓存状态（最近 → 最久）
put(1,1), put(2,2) - 2:2 → 1:1
get(1) 1 1:1 → 2:2
put(3,3) - 3:3 → 1:1，key 2 被淘汰

考察知识点

HashMap

用 key 直接定位链表节点，保证查找、更新、删除入口都是 O(1)。

双向链表

把最近使用的节点移动到头部，把最久未使用的节点放在尾部附近。

数据结构设计

这题不是单个 API 调用，而是维护多个操作之间一致的数据结构状态。

官方方案对照

LeetCode 官方题解使用的是 哈希表 + 双向链表：哈希表负责通过 key 找节点，双向链表负责维护最近使用顺序。这个页面讲解的也是同一个最优方案。

为什么等价于官方最优解

每个 key 在 HashMap 中对应一个链表节点。访问或写入成功后，把节点移到链表头部；容量超限时，删除尾部节点，并同步从 HashMap 删除。

为什么不用单 HashMap

HashMap 能 O(1) 找 key，但无法 O(1) 知道谁最久未使用；单链表能维护顺序，但无法 O(1) 删除任意节点。

核心思路

HashMap 找节点
访问后移到头部
容量满删尾部
Map 和链表同步

可以把双向链表看成一条队伍：队头是最近使用，队尾是最久未使用。为了避免处理空链表、头尾删除等边界，常用两个哨兵节点 head 和 tail。

Java 参考实现

import java.util.HashMap;
import java.util.Map;

class LRUCache {
    // 双向链表节点：key 用于淘汰时回删 HashMap，value 是缓存值。
    private static class Node {
        int key;
        int value;
        Node prev;
        Node next;

        Node() {
        }

        Node(int key, int value) {
            this.key = key;
            this.value = value;
        }
    }

    private final int capacity;

    // cache 负责 O(1) 通过 key 找到链表节点。
    private final Map<Integer, Node> cache = new HashMap<>();

    // head 和 tail 是哨兵节点，不存真实数据。
    // head.next 永远是最近使用的节点，tail.prev 永远是最久未使用的节点。
    private final Node head = new Node();
    private final Node tail = new Node();

    public LRUCache(int capacity) {
        this.capacity = capacity;

        // 初始化空链表：head <-> tail。
        // 有哨兵后，插入和删除时不用单独判断空链表或首尾节点。
        head.next = tail;
        tail.prev = head;
    }

    public int get(int key) {
        Node node = cache.get(key);
        if (node == null) {
            return -1;
        }

        // 被访问过的 key 变成最近使用，所以移动到链表头部。
        moveToHead(node);
        return node.value;
    }

    public void put(int key, int value) {
        Node node = cache.get(key);
        if (node != null) {
            // 已存在的 key 只更新值，不创建新节点；随后刷新使用顺序。
            node.value = value;
            moveToHead(node);
            return;
        }

        // 新 key 同时写入 HashMap 和双向链表。
        // 链表头部代表最近使用，因此新节点直接放到 head 后面。
        Node created = new Node(key, value);
        cache.put(key, created);
        addToHead(created);

        if (cache.size() > capacity) {
            // 超容量时淘汰 tail 前面的真实节点，也就是最久未使用节点。
            // 删除链表节点后，必须同步删除 HashMap 中的 key。
            Node removed = removeTail();
            cache.remove(removed.key);
        }
    }

    private void moveToHead(Node node) {
        // 移动节点分两步：先从原位置摘掉，再插入到头部。
        removeNode(node);
        addToHead(node);
    }

    private void addToHead(Node node) {
        // 把 node 插入到 head 和原 head.next 之间。
        node.prev = head;
        node.next = head.next;
        head.next.prev = node;
        head.next = node;
    }

    private void removeNode(Node node) {
        // 因为有哨兵节点，node 一定有 prev 和 next，直接改两侧指针即可。
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }

    private Node removeTail() {
        // tail.prev 是当前最久未使用的真实节点。
        Node node = tail.prev;
        removeNode(node);
        return node;
    }
}

变量和方法职责

成员/方法 作用 易错点
cache key 到 Node 的映射 删除尾节点时必须同步删除 map 中的 key
head.next 最近使用的真实节点 访问或更新 key 后要移动到这里
tail.prev 最久未使用的真实节点 容量超限时淘汰它，而不是刚插入的节点
moveToHead 先摘除节点，再插入头部 不能只改一边指针，双向关系要成对维护

易错点

  1. get 后忘记刷新使用顺序：命中 key 后必须移动到头部。
  2. put 更新旧 key 时只改 value：更新也算使用，需要移动到头部。
  3. 链表删了，Map 没删：之后 get 会拿到已经不在链表中的旧节点。
  4. 指针更新顺序错：双向链表插入/删除要同时维护前驱和后继。
  5. 不用哨兵节点：头尾边界会变复杂，容易漏掉容量为 1 的情况。

复杂度分析

时间复杂度：O(1)

get 和 put 都只做 HashMap 查询、常数次链表指针修改。

空间复杂度：O(capacity)

最多保存 capacity 个节点，HashMap 和链表节点数量同阶。
